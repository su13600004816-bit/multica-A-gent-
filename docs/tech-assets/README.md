# 技术储备库（tech-assets）

四个**独立技术**，各成自包含体，给"未来新站"一键迁移用，同时当下 multica 也在用。**每个互相独立，绝不合到一起**——照片是照片、记忆是记忆、看门狗是看门狗、画布是画布。

| 技术 | 目录 | 当前状态 |
|---|---|---|
| 图片压缩 | `image-compression/` | ✅ 已上线 multica + 真测 PASS |
| 记忆储存(全局四层) | `memory-store/` | ✅ 已上线 multica + generator=deepseek 实测 |
| 看门狗 | `watchdog/` | ✅ 仓库内三层在产；外层 line-watchdog 套件已收录 |
| 画布编排 | `canvas-orchestration/` | 🚧 改进中（苏总缓）；squad-canvas 页在产 |

## 每个目录的结构
- `SPEC.md` —— 完整说明书：作用/原理、本包文件、依赖、对外接口、**迁移接入新系统、同步、改进指南、删除卸载**、验证方法、已知点、当前部署状态。
- `package/` —— 自包含可整块搬运的真实代码副本（独立于任何 multica 分支）。

## 维护协议（苏总硬规：跟新系统同步/改进/删除）
1. **独立**：四块各自维护，不准为了部署把它们合进同一分支造耦合。多 multica 部署是单体现实（前端一镜像/后端一镜像），但储备包按技术独立存。
2. **同步**：multica/老站里的实现有改进 → 同步回对应 `package/` 并在该 SPEC 第 6 节登记。本库是储备 SSOT。
3. **改进**：见各 SPEC 第 7 节"改进指南"。
4. **删除**：见各 SPEC 第 8 节"删除/卸载步骤"，保证干净移除无遗留。
5. **登记**：任何同步/改进/删除都在对应 SPEC + 主台账 `/opt/multica/TECH_ASSETS_MIGRATION_LEDGER.md` 留痕。
